Ma Paddy - Database Blueprint (MVP)

Architectural Design for Civic-Tech Mental Wellness Web App

Role: Senior Product Architect & Civic-Tech Developer

1. Core Database Tables

Table: users

Supports both anonymous and registered flows.

Field Type Status Notes
id UUID Required Primary Key.
device_identifier String Required Browser fingerprint/Unique ID for anonymous sessions.
auth_provider_id String Optional For linked accounts (Email/Google).
nickname String Optional User-chosen name (never use real names).
language_pref Enum Required en, fr, pidgin.
created_at Timestamp Required Used for onboarding analytics.

Table: mood_logs

Field Type Status Notes
id UUID Required Primary Key.
user_id UUID Required Foreign Key.
mood_score Integer Required 1 (Terrible) to 5 (Great).
emotions JSON/Array Optional Tags like ["lonely", "stressed"].
logged_at Timestamp Required Defaults to current time.

Table: chat_sessions

Groups messages into logical daily or topic-based sessions to reduce data load.

Field Type Status Notes
id UUID Required Primary Key.
user_id UUID Required Foreign Key.
session_summary Text Optional AI-generated summary for quick history recall.
last_activity Timestamp Required For sorting history.

Table: messages

Field Type Status Notes
id UUID Required Primary Key.
session_id UUID Required Foreign Key.
sender_type Enum Required user or paddy (AI).
content Text Required Encrypted at rest.
created_at Timestamp Required Used for ordering.

Table: journals

Field Type Status Notes
id UUID Required Primary Key.
user_id UUID Required Foreign Key.
content Text Required Encrypted user reflections.
sentiment_tag String Optional Detected sentiment (e.g., "Reflective").
created_at Timestamp Required Timestamp.

2. Relationships & Entity Diagram

USERS (1) <---> (N) MOOD_LOGS
USERS (1) <---> (N) JOURNALS
USERS (1) <---> (N) CHAT_SESSIONS
CHAT_SESSIONS (1) <---> (N) MESSAGES

Note: All relations use Cascading Deletes. If a user chooses to delete their account, all associated mental health data is wiped immediately.

3. Supporting Anonymous Users Safely

  • LocalStorage Token: When a user starts anonymously, generate a UUID and store it in the browser's localStorage. This maps to the device_identifier in the users table.
  • Graceful Transition: If the user decides to create an account later, we update the users record with their credentials, keeping their chat and mood history intact.
  • Session Expiry: Anonymous sessions can be flagged for archival after 90 days of inactivity to protect privacy and save space.

4. Trust & Safety: What NEVER to store

  • NO Precise Geo-location: Only store general region (e.g., Littoral, Centre) if necessary for local resource guidance.
  • NO Real Names: Encourage the use of "Paddy Names" or nicknames.
  • NO Plaintext Content: Chat messages and Journal entries must be encrypted at rest (AES-256).
  • NO Phone Numbers/ID Cards: This is a companion app, not a clinical health record.

5. Scalability & Low-Data Optimization

  • Pagination Message Batching: Only load the last 10 messages of a chat session. Older messages are loaded only on "scroll up."
